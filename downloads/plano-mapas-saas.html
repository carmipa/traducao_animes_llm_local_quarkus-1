00Resumo

A ideia em seis linhas

Hoje você tem um motor de mapas 100% próprio na sua VPS: geocodificação (endereço → coordenada), roteirização (rota de carro + ETA) e CEP — sem pagar Google/Mapbox e sem depender de API de terceiros. O site de delivery já conversa com esse motor só por variável de ambiente (OSRM_URL, NOMINATIM_URL).

O plano: quando você vender o site para outros restaurantes, você não entrega o motor — ele vira uma API central sua (SaaS), e cada cliente recebe só uma URL + um token. Quem não pagar, você corta o token. Isso protege seu ativo mais caro (o motor), isola recursos (mapas puxam RAM/CPU) e permite cobrar por uso — assinatura ou créditos pré-pagos.

01Fundamentos

O que é o "motor de mapas" — as 3 peças

Um serviço de delivery precisa responder 3 perguntas de mapa. Você já resolve as três localmente:

PerguntaPeçaEngineEstado
"Onde fica esse endereço?"
endereço → lat/lng		GeocoderNominatim (OSM de SC)soberano
"Qual o caminho e o tempo?"
A→B → linha azul + ETA		RouterOSRM (OSM do Sul)soberano
"Que endereço é esse CEP?"CEPViaCEP (público)no ar
"Onde fica, mesmo digitado errado?"Geocode fuzzyPhotonopcional

Por que isso vale ouro

Google Maps cobra ~US$5 / 1.000 chamadas de geocode e ~US$5 / 1.000 de rota. Um delivery com movimento faz milhares de chamadas/dia (cada checkout geocodifica, cada rastreamento recalcula rota). Rodando você mesmo, o custo marginal por chamada é ~zero — só CPU/RAM que já é sua. É exatamente esse delta que você revende.

Como o site fala com o motor

No docker-compose.yml, o serviço app recebe só URLs — nunca fala com Google:

docker-compose.yml · service app
# roteirização e geocode soberanos (self-hosted)
- OSRM_URL=http://osrm:5000
- NOMINATIM_URL=http://rede-restaurantes-nominatim:8080
# PHOTON_URL não setado → fuzzy desligado (fallback Nominatim)

Comunicação por rede Docker interna — os engines não estão expostos publicamente. É exatamente o modelo que o SaaS vai usar.

Convivência com o ARK

A mesma VPS roda o servidor de ARK (faminto de RAM). Existe o guardião nominatim-ramguard.sh que pausa/retoma o Nominatim sob pressão de RAM. Isso é um sintoma: para um SaaS de verdade, o motor deve migrar para VPS dedicada (Roadmap · Fase 3).

Cadeia de geocode (robusta, estilo iFood/Uber)

Função central _geocode_endereco(...) em app.py:7954 — tenta em cascata (para na primeira que acerta):

  1. Photon (se PHOTON_URL) com a string completa — tolera erro de digitação.
  2. Nominatim estruturado (street/city/state/country) — bem mais preciso que texto livre.
  3. Canonicalização por ViaCEP — logradouro/cidade/UF oficiais do CEP e re-tenta.
  4. Nominatim só por CEP (centro do CEP) — último recurso.
  5. Texto livre e por bairro — rede de segurança.

Se NOMINATIM_URL não estiver setado, cai no Nominatim público — fail-open: o site nunca quebra por falta do motor local, só fica menos soberano.

04Estratégia

O plano SaaS — de "motor interno" para "API que se vende"

Por que separar — os 3 motivos

  1. Proteção de IP — o motor é o ativo caro. Se ele for junto no pacote, o cliente replica e para de pagar. Sem o motor, o site é uma casca que só funciona apontando pra sua API. Não pagou → revoga o token → o mapa dele morre.
  2. Isolamento de recursos — OSRM/Nominatim puxam RAM/CPU. Na máquina do cliente, um pico de mapa derruba o site dele (e a culpa é sua). Centralizado, você controla a capacidade.
  3. Multi-tenant + escala — 1 motor serve todos os clientes. Importa os dados uma vez; cada novo cliente é só mais um token. Margem cresce sem custo proporcional.

O que muda na entrega ao cliente

compose ENTREGUE ao cliente — sem engines, só o token
services:
  app:
    environment:
      - MAPS_API_URL=https://mapas.suaempresa.com     # sua API central
      - MAPS_API_TOKEN=rk_live_casa42_a1b2c3...        # token do cliente
      # NÃO existe serviço osrm/nominatim/photon aqui.

Regra de ouro

O compose e a imagem entregues ao cliente não podem conter os serviços osrm/nominatim/photon nem os dados osrm-data/ / nominatim-pgdata/. Só a URL + token.

Arquitetura alvo

gateway central + engines atrás + clientes por token
   ┌──────────────────── SUA VPS CENTRAL ────────────────────┐
   │  ┌─────────────┐  valida token, mede uso, rate-limit,   │
HTTPS│ │  GATEWAY     │  cacheia, esconde qual engine usa      │
+token│►│  /v1/*      │──┬─► OSRM (Sul)                         │
   │  └──────┬──────┘  ├─► Nominatim (SC → depois BR)          │
   │  ┌──────▼──────┐  ├─► Photon (fuzzy, opcional)            │
   │  │ tokens_db    │  └─► ViaCEP (proxy externo)             │
   │  │ planos/quota │                                         │
   │  └─────────────┘                                          │
   └──▲──────────────▲──────────────▲───────────────────────┘
      │ rk_live_A    │ rk_live_B    │ rk_live_C
   [Site delivery A][Site delivery B][Site delivery C]  (VPS dos clientes)

O Gateway é uma peça nova e fina (não é o app de delivery): valida o token, checa quota/rate-limit, encaminha para a engine certa, cacheia, incrementa o uso e esconde qual engine está por trás — você pode trocar OSRM por outro sem o cliente saber.

06Receita

Modelo de negócio — como cobrar pelos tokens

Dois modelos, e o gateway suporta os dois ao mesmo tempo (assinatura com quota + saldo pré-pago de excedente):

Assinatura mensal (recorrência — base recomendada)

PlanoPreço/mêsCota mensalRateExtras
TrialR$ 01.000 req5/s30 dias
StarterR$ 4920.000 req10/sgeocode + rota
ProR$ 149100.000 req20/s+ matriz multiparada
PremiumR$ 399500.000 req40/s+ tiles próprios
ExcedenteR$ 6 / 1k——o que passar da cota

Referência de preço

Números ilustrativos — ajuste pela sua margem. O mesmo volume no Google custaria ~US$5/1.000 (geocode) + ~US$5/1.000 (rota). Você pode precificar em 10–20% do Google e ainda ter margem gigante, porque seu custo marginal é ~zero.

Pré-pago (créditos / "tokens de verdade")

Cliente compra um pacote de créditos (ex.: 50.000 por R$ 200). Cada chamada consome 1 crédito (rota multiparada custa 2–3). Zerou, para até recarregar. Bom para volume irregular ou quem não quer recorrência.

Como o dinheiro entra

Você já tem Mercado Pago e Asaas no projeto. Reaproveite:

  • Assinatura → recorrência do MP → webhook ativa/renova a quota do token do mês.
  • Pré-pago → link de pagamento por pacote → webhook → /admin/tokens/:id/recarga.
  • Inadimplência → webhook de falha/cancelamento → ativo=false → próximas chamadas retornam 403 token_revogado. Corte automático.

07Dados

tokens_db — esquema mínimo

SQL · SQLite no MVP, Postgres depois
CREATE TABLE tokens (
  id             SERIAL PRIMARY KEY,
  chave_hash     TEXT NOT NULL,      -- só o sha256 da chave, nunca em claro
  prefixo        TEXT NOT NULL,      -- "rk_live_casa42" p/ exibir no painel
  cliente        TEXT NOT NULL,
  plano          TEXT NOT NULL DEFAULT 'trial',
  quota_mes      INTEGER NOT NULL DEFAULT 1000,
  rate_limit_rps INTEGER NOT NULL DEFAULT 5,
  saldo_prepago  INTEGER NOT NULL DEFAULT 0,
  ativo          BOOLEAN NOT NULL DEFAULT true,
  validade       DATE,               -- fim do ciclo pago
  criado_em      TIMESTAMP DEFAULT now()
);
CREATE TABLE uso_mensal (
  token_id INTEGER REFERENCES tokens(id),
  ciclo    TEXT,                      -- "2026-07"
  usados   INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (token_id, ciclo)
);

Rate-limit: token bucket em memória (ou Redis se multi-processo). Cache de geocode/rota: herdar dos caches atuais.

09Empacotamento

Migração do compose entregue — o passo que protege seu IP

  1. Remover os serviços osrm, nominatim, photon do compose.
  2. Não incluir as pastas osrm-data/ e nominatim-pgdata/ (GB de dado + seu trabalho de import).
  3. Trocar as envs das URLs internas por MAPS_API_URL + MAPS_API_TOKEN.
  4. Garantir que nenhum .py contenha URL/porta interna dos engines (só o shim lendo as duas envs).
diff · envs do app entregue
- OSRM_URL=http://osrm:5000
- NOMINATIM_URL=http://rede-restaurantes-nominatim:8080
+ MAPS_API_URL=https://mapas.suaempresa.com
+ MAPS_API_TOKEN=rk_live_<cliente>_<segredo>

Checklist anti-vazamento (no build de entrega)

grep -rInE 'osrm|nominatim|photon|8080|:5000' . no pacote do cliente não pode retornar nada além do maps_client.py lendo MAPS_API_*.

10Guarda

Segurança, licença e cuidados

  • Token nunca em claro — guarde sha256(chave). Mostre no painel só o prefixo.
  • HTTPS obrigatório no gateway (o token viaja no header). Caddy já faz TLS — reaproveite.
  • Rate-limit + quota desde o dia 1 — senão um cliente (ou um bug do site dele) te derruba a infra.
  • Rede interna para os engines — só o gateway os enxerga; nunca porta pública.
  • Licença OSM / ODbL — os dados são do OpenStreetMap. Você pode vender acesso a um serviço, mas precisa atribuir "© colaboradores do OpenStreetMap" onde o mapa aparece. Vender serviço/uso = ok; o cuidado é atribuição + não vender "os dados OSM" como se fossem seus.
  • Fail-open é bom p/ você, ruim p/ cliente — no shim do cliente, se o token falhar, retorne erro claro; não vaze pro Nominatim público (perde controle/medição).

12Handoff

Prompt de construção (copiável)

Cole isto num dev ou numa IA para construir o serviço mapas-gateway. É a especificação executável de tudo acima.

prompt · mapas-gateway
Construa um microserviço HTTP "mapas-gateway" (Python, Flask + gunicorn, Docker),
FACADE de uma API de mapas soberana vendida por assinatura/token. Fica na frente
de três engines self-hosted numa rede Docker interna:
  - OSRM       http://osrm:5000        (formato OSRM /route/v1/driving)
  - Nominatim  http://nominatim:8080   (formato Nominatim /search)
  - ViaCEP     https://viacep.com.br/ws/<cep>/json
  - Photon     $PHOTON_URL (opcional)  (formato Photon /api)

REQUISITOS:
1. Auth por token no header "Authorization: Bearer rk_live_...". Guardar só sha256.
   Banco tokens(id, chave_hash, prefixo, cliente, plano, quota_mes, rate_limit_rps,
   saldo_prepago, ativo, validade, criado_em) + uso_mensal(token_id, ciclo, usados).
   SQLite no MVP; Postgres depois.
2. Middleware por request: validar token → checar ativo (senão 403 token_revogado) →
   rate-limit token-bucket por token (senão 429) → checar quota do ciclo YYYY-MM
   (senão 402 quota_excedida) → processar → incrementar uso → responder com headers
   X-Quota-* e X-RateLimit-*.
3. Endpoints v1:
   POST /v1/geocode {endereco,numero,bairro,cidade,uf,cep,fuzzy?}
        -> {ok,lat,lng,precisao,fonte,cache}. Cascata Photon(se fuzzy)→Nominatim
        estruturado→canonicaliza via ViaCEP→CEP→texto livre. Fail: 200 {ok:false}.
   GET  /v1/cep/<cep> -> {ok,logradouro,bairro,cidade,uf} (proxy ViaCEP, cacheado)
   POST /v1/route {origem,destino,paradas?,geometria}
        -> {ok,coords[[lat,lng]],distancia_m,duracao_s,eta_texto,cache}.
        OSRM /route/v1/driving/{lng,lat};...?overview=full&geometries=geojson,
        converte [lng,lat]->[lat,lng]. Cache TTL 25s por origem/destino arred. 4 casas.
   POST /v1/matrix {pontos[]} -> {duracoes_s[][],distancias_m[][]} (OSRM /table) [fase2]
   GET  /v1/usage -> {plano,ciclo,quota_mes,usados_mes,restante,rate_limit_rps,...}
4. Admin (header X-Master-Key = $MASTER_KEY):
   POST /admin/tokens {cliente,plano,quota_mes,rate_limit_rps} -> gera rk_live_<slug>_<rand>
        (devolve a chave UMA vez).
   GET /admin/tokens ; POST /admin/tokens/<id>/recarga ; DELETE /admin/tokens/<id>
   GET /admin/usage?mes=YYYY-MM -> consumo por cliente (base da fatura).
5. Envelope de erro único: {ok:false, erro:{codigo,mensagem}} com HTTP 400/401/402/403/429.
6. Config por env: OSRM_URL, NOMINATIM_URL, PHOTON_URL(opc), MASTER_KEY, DATABASE_URL.
7. HTTPS terminado por Caddy. Engines NUNCA ganham porta pública.
8. Pronto p/ webhook de pagamento (MP/Asaas) chamar /admin/tokens/<id>/recarga (renova)
   ou setar ativo=false (corta inadimplente).
Entregue: app.py do gateway, Dockerfile, trecho de docker-compose, e um maps_client.py
(shim) p/ o delivery consumir via MAPS_API_URL + MAPS_API_TOKEN.
